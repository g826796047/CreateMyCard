# Harmony 卡片分层设计系统方案

## 1. 背景、目标与非目标

本方案为主 Agent 提供一条独立的 HarmonyOS A2UI Form 卡片生成路径。它模拟人类设计卡片时的决策顺序：先理解需求特征，再选择布局、模块、元素和内容，最后由确定性装配器生成 `genui` 与 `cardspec`。

目标是把模型自由度约束在经过校验的布局几何、模块 footprint、元素尺寸档位和内容槽位内，同时允许同一布局使用不同兼容模块，不再把完整成品模板作为唯一选择单位。

本方案不负责云侧 artifact，不调用微服务，也不调用或读取另外两个生成 Skill。它不支持完整 App 页面、复杂表单、长报告、任意尺寸或未声明组件。

## 2. 三个 Skill 的独立关系

| Skill | 方案 | 输出 |
| --- | --- | --- |
| `harmony-card-generation-online` | 云侧工具编排 | artifact 链接和状态说明 |
| `harmony-card-generation-offline` | 通用规则自由生成 | DSL 与 CardSpec |
| `harmony-card-template-generation` | 分层设计系统与确定性装配 | DSL 与 CardSpec |

三个 Skill 互不调用、互不兜底。模板生成 Skill 只通过显式 `$harmony-card-template-generation` 使用，并保持 `allow_implicit_invocation: false`。

## 3. 五层设计模型

```text
用户 query
 -> Feature Profile：服务对象、主问题、内容优先级、数据、动作、素材、尺寸和风格
 -> Layout：卡片主模块的方向、数量、父子关系、宽高和间距
 -> Module：每个布局区域承担的信息职责和内部结构
 -> Element：模块内 Text/Image/Progress/Button/Divider 的规格和尺寸档位
 -> Content：元素展示值、DataModel 路径、素材和合法 onClick
 -> Design Plan
 -> 确定性装配 genui + CardSpec
```

`2x2` 最多三个顶层模块，`2x4` 最多四个。模块可以包含内部信息组，但不能借内部嵌套绕过单一主问题、动作数或能力数限制。

## 4. 资产职责

- `assets/templates/index.json`：设计系统入口，登记 schema、catalog、布局、模块、元素、风格和 CardSpec 基础骨架。
- `assets/templates/schemas/`：约束 Feature Profile、Design Plan、Layout、Module 和 Element Catalog。
- `assets/templates/layouts/catalog.json`：维护十二个固定布局及 region 几何。
- `assets/templates/modules/catalog.json`：维护模块职责、精确 footprint 变体、组件树和内容槽位。
- `assets/templates/elements/catalog.json`：维护组件 token、字号、图标、进度和按钮尺寸档位。
- `assets/templates/cardspec/base.json`：维护静态 CardSpec 基础字段。
- `scripts/tests/fixtures/`：保存由原十三个成品模板抽象出的设计计划和黄金产物，仅用于参考与回归。

原十三个模板不再拥有运行时模板 ID，也不是最终组合白名单。它们只用于证明新的布局、模块和元素抽象能够覆盖既有核心构图。

## 6. Feature Profile 与 Design Plan

Feature Profile 固定描述：唯一服务对象、唯一主问题、目标尺寸、主视觉语义、内容项、优先级、动作、数据需求、素材需求和风格意图。每个内容项必须有稳定 ID，并标为 `mustKeep` 或 `shouldKeep`。

Design Plan 固定记录：Feature Profile、layout ID、style profile、每个 region 的 module/variant、元素 token、内容映射、初始 DataModel、CardSpec 数据绑定、事件、素材和降级记录。

所有 `mustKeep` 内容必须恰好映射一次。`shouldKeep` 可以删除，但必须记录降级原因。不得新增 Feature Profile 中不存在的第二服务对象或第二主问题。

详细字段见 [分层模板契约](references/template-contract.md)，选择流程见 [路由与风格](references/routing-and-style.md)。

## 7. 模块、元素与内容边界

模块声明一个或多个精确 footprint 变体。变体包含抽象组件树、本地组件 ID、允许的 element token、内容槽位、字符预算和可选事件目标。模块本地 ID 在装配时统一命名为 `regionId__moduleId__localId`。

Element Catalog 维护有限规格。Agent 只能选择模块节点声明的 token，不得在 Design Plan 中直接写宽高、字号或组件类型。容器宽高来自布局和模块变体，内容元素尺寸来自 element token 与模块节点的受控样式。

内容槽位声明绑定方式：`static`、`expression` 或 `path`。动态路径必须能由初始 DataModel 或 CardSpec `writeResultTo + outputSchema` 推导。删除内容时同步清理元素、DataModel、事件和无用数据绑定。

## 8. 数据、事件、素材与 CardSpec

`2x2` 最多一个数据能力，`2x4` 最多两个。能力必须来自 [数据能力索引](references/capability/data-capability/index.md)，未声明能力只能静态降级。

点击事件只写 DSL `onClick`，不写入 CardSpec；事件必须来自 [点击事件能力](references/capability/event-capability/click-event.md)。素材只使用 [素材库](references/design/asset-library.md) 声明的资源路径。

CardSpec 只包含静态 `title`、静态 `description`、`suggestSize` 和有效 `dataBindings`。协议 catalog 统一为 `ohos.a2ui.extended.catalog.form`。

## 9. 装配、校验与降级

主 Agent 先生成 Design Plan，校验通过后调用 `scripts/assemble_card.py`。装配器按布局创建 root 和 region，按模块变体展开组件树，按 element token 注入组件规格，最后生成三行 JSONL 与 CardSpec。

`validate_card.py --design-plan` 同时检查协议合法性和产物是否与 Design Plan 的确定性装配结果一致。Agent 不直接修改装配后的 DSL；首次失败时只能修改 Design Plan 并重新装配一次。

无匹配时依次删除 `shouldKeep`、选择同尺寸更简单布局、尝试 `2x4`。仍不成立时要求简化需求，不调用其它 Skill，也不退回自由布局。

## 10. 测试、扩展与同步

测试覆盖十二个布局、模块兼容矩阵、元素尺寸、十三组参考设计、四种风格、数据、事件、素材和所有主要失败场景。所有资产必须通过 `validate_templates.py --all --strict`，最终产物必须通过带 `--design-plan` 的严格校验。

新增布局、模块、元素或设计规则时，先更新本文件，再同步 schema、catalog、references、脚本和测试。普通生成任务不读取本文件。

共享 A2UI 协议和端侧能力边界服从项目 `docs/云侧方案设计.md`；发生冲突时先确认权威方案，再修改本 Skill。
